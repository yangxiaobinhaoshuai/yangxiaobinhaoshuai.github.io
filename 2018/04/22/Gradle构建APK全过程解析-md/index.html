Android Gradle plugin 源码分析

Gradle 是一个基于 groovy 但可以兼容任何基于 JVM 语言的强大构建工具。因为 DSL 和 Gradle 声明周期的灵活性，使得构建项目脚本配置不在冗余，且提供了例如 BuildType、productFlovor 等构建模式，让项目能在多种环境构建之间自由切换；而且多种多样的 Gradle plugin 也让项目构建如虎添翼。

而在 Android 中，为了更方便应用开发者，Google 推出了 Android gradle plugin 在 Gradle 许多原有基础构建功能上，添加了定制化功能；也是我们 build.gradle 许多可配置节点的如 Android {} 的来源。

至于插件本身，就是默认配置i在 project 级 build.gradle 中的 classpath 'com.android.tools.build:gradle:3.1.1' ， 我们平时所说的 Android Studio 2.3 和 3.0 其实指的就是 Gradle 中 Android plugin 的版本。

Gradle 构建 APK 流程

为了更好说明 Gradle 构建过程，简单介绍下 Gradle 相关知识：

Gradle 插件基础功能

  1. Gradle 基础 Task 有 assemble（生成产出物，如 jar、war 等）、clean（清除产出物）。
  2. Java 插件基于基础 Task 提供了更强大的实现
    1. assemble : 输出 jar 包；
    2. check ：运行单元测试和集成测试；
    3. build ： asseble + check ；
    4. clean ： 删除输出目录文件；
  3. Android Gradle plugin 又基于 Java 插件为基础 Task 提供了 Android 定制化功能和增加了若干新 Task
    1. assemble : 输出 apk
    2. check : 运行 Lint 检测，一旦检测异常停止构建过程；
    3. build ： assemble + check
    4. clean : 删除输出目录

Gradle 的生命周期

任何 Task 运行都会有三个阶段 ：Initialization —> Configuration —> Execution

APK 编译过程

介绍 Android plugin 工作流程前我们先了解下 Android apk 编译过程，上一张老生常谈的图：

编译过程分七步：

  1. aapt 编译资源文件，生成 R 文件
  2. 编译 aidl 文件
  3. 编译 java 代码为 class
  4. dx 编译 class 为 dex
  5. 打包 apk
  6. sign apk
  7. zipalign 对齐处理 apk

APK 编译流程对应 Gradle tasks

Terminal 执行：./gradlew assembleDebug --drt-run ，dry run 表示只显示 build 过程 task name 不做实际构建工作， task 后面的 SKIPPED 也表示该 task 并没有被实际执行。
