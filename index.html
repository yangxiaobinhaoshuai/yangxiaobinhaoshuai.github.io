关于65535的问题

发表于 2018-01-17
避免64官方文档
Android 的瓶颈
  1. 在android 源码中搜索 报错关键字，Too many field references: 131000; max is 65536. 等即可。 原因大概是dalvik 给method id 分配的大小就是16位，所以上限就是2的16次方，65536，从0开始就是65535.
  2. 65535代表单个dex可调用的引用总数。其实不仅仅是method，feild，class，也有这限制，只不过方法id更容易达到上限而已。
  3. 所以65535问题是无法从根本上解除的，我们只能通过各种手段来规避这个上限，而google给出的手段是multidex分包
Multidex
  1. minSDK5.0之前
    1. dalvik 模式：Android使用dalvik来运行应用。
      1. JIT编译：热点代码会被转化成本地机器码来提高运行效率
    2. 运行在api21 以下，Android studio 会停用InstantRun
    3. 需要compile 分包执行库
    4. 应用配置：
      1. build.gradle android节点—> defaultConfig节点—> multiDexEnabled true 开启multidex
      2. extends MultiDexApplication ，记得注册Applicaton
        1. 如果没法继承 MultiDexApplication， 覆写 attachBaseContext 调用MultiDex.install(this)
  2. minSDK5.0之后
    1. art 模式：采用应用安装就变成本地机器码
      1. 优点：运行更快
      2. 缺点：安装慢，apk体积更加10%～20%
    2. Android 已经支持了APK加载多个dex文件，不需要分包执行库
    3. 在应用安装的时候会执行预编译，将多个dex编译成单个odex
    4. 调试的时候，InstantRun 会自动进行分包，但是Release 的时候还是要手动分包，因为InstatnRun 只用于Debug 模式。
    5. 应用配置：
      1. build.gradle android节点—> defaultConfig节点—> multiDexEnabled true 开启multidex
    6. Google 官方支持的MultiDex jar包 从api 21 就在build tools 里了
避免64k措施
  1. 不要为了几个方法而依赖庞大体积的库
  2. proguard 启用代码压缩
MultiDex 局限性

  1. 如果辅助Dex过大，可能导致应用启动后ANR。 解决办法：混淆。

  2. 启动报错，在低版本上会发生。解决办法也是压缩代码。

  3. OOM，低版本和高版本都可能发生。

  4. NoClassDefFoundError，代码路径可见性低的时候，比如动态加载，或者依赖复杂的时候，或者调用JNI，这些情况这些类可能被分到其他Dex，主dex启动的时候会报错。

    1. 解决办法：手动添加到主Dex必需项

      1. MultiDexKeepFile

        1. 可以手动创建一个multidex-config.txt文件 然后编辑

        4. 这里注意txt 的相对路径，以上用户txt是和build.gradle在同一级目录下才可以的

      2. MultiDexKeepProguard

        1. 与proguard 格式一样。

        2. 手动创建 multiDexKeepProguard.txt ，使用proguard 语法编辑

        4. 如果想指定包下所有类都在主Dex

        7. 还是注意txt 的目录层级问题

      3. 使用了multidex后，gradle 在build 的时候的步骤是

        1. 根据manifest 和build.gradle 的依赖关系 把四大组件和support包等关键信息写在manifest_keep.txt 在build目录下
        2. 根据proguard 规则来优化，将没有用到的类删除，生成ComponentClasses.jar
        3. 扫描 componentClasses.jar 创建maindexlist.txt 里面包含main dex 所有所需的类。
          1. 这里列出的类并不全
          2. 这个文件是可编辑的，如果要添加必要类，直接按语法添加即可
            1. 如果类并不存在，gradle 会忽略
            2. 如果类重复，则只取其中一个

  5. 应用安装后第一次启动会非常慢，android 会用DexOpt 生成odex ，这一步比较耗时，而非安装后第一次启动，只需加载odex 这个速度很快

  6. 这里注意： multiDexKeepFile file ‘multidex-config.txt’，会报错，要用multiDexKeepProguard file(‘multdex-proguard.pro’)

  7. ​

优化开发建议
  1. multiDex 会增加build 时间，因为build过程会决定哪些类在主dex 中，要移除哪些类。
  2. 通过配置productFlavors 配置两个minSDK 来减少调试的时候的时间
    1. api 21 以下
      1. 为了适配更多的机型
    2. api 21 以上
      1. 因为api21 以上的ART 模式会更快的生成Dex文件
      2. 执行pre-dexing ，将每个应用模块和每个依赖项编译成单个dex
      3. 将每个dex加入apk，且不做修改，即不混淆，因为minifyEnable为false
      4. 模块的dex 不会执行合并，避免了确定主dex哪些类而耗费时间
      5. 增量构建，只有修改过的dex会被重新编译。
DexOptions
  1. –multi-dex：多 dex 打包的开关。
  2. （可选）–main-dex-list=：参数是一个类列表的文件，在该文件中的类会被打包在第一个 dex 中。
  3. （可选）–minimal-main-dex：只有在–main-dex-list 文件中指定的类被打包在第一个 dex，其余的都在第二个 dex 文件中。
  4. （可选）–set-max-idx-number： 可以用于设置 dx 打包时每个 dex 最大的方法数
  5. ​
Dex 加载
  1. Android 系统应用启动的时候只加载主Dex，其他dex 需要在应用启动后动态加载安装。
  2. MultiDexApplication 启动—> MultiDex install —> install SecondDexes —>support lib install —> expandeFileArray —> makeDexElements—> DexPathList—> loadDexFile
  3. 冷启动容易触发ANR：
    1. 解压apk
    2. 加载主dex
    3. 反射加载其他dex
      1. 异步加载其他dex
      2. 人工干预dex拆分，使每一个dex都在合理范围内
        1. 通过gradle 中定义一个task 来干预dex产生
MultidDex 原理
  1. multidex 会在application 初始化之后，检查版本是否支持multidex ，classes2.dex 是否需要安装
  2. 如果需要安装则会从apk中解压出classes2.dex 将其拷贝到应用沙盒目录下
  3. 通过反射将classes2.dex 注入到当前的classloader中
Dex 解决问题
  1. 能正常分包打出多个dex
  2. 可以指定每个dex 大小
  3. 可以加载多个dex
  4. 可以指定主dex 包含哪些类
